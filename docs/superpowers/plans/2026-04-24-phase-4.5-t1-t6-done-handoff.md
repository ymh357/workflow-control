### T6 — AI-driven propose_pipeline_fix patch synthesis
SHAs: `d13a009`, `f67466c` (2 commits)

  => Promise<IRPatch | null>`. Tests inject stubs; production uses
  `createClaudeSdkPatchSynthesizer` with real Claude Agent SDK
  query().
- `proposePipelineFixWithAi`: wraps rule-based layer; for every
  non-info suggestion calls synth in parallel. Errors swallowed,
  null responses leave proposedPatch undefined.

  safe-range fields (budget / reads / writes) explicitly out of scope
  until validated need.
- MCP: `propose_pipeline_fix` tool gains optional `aiPatch: boolean`
  input (default false). Default behaviour unchanged — rule-only,
  instant, no API cost. `aiPatch=true` burns API tokens per non-info
  suggestion.
- Claude SDK synthesiser handles: JSON with/without code fence,
  NO_PATCH sentinel, malformed JSON, wrong op type, wrong target
  stage, stream errors.
- Tests: 5 wrapper integration + 9 synth unit = 14 new cases.
- Roadmap §6.4 propose_pipeline_fix flipped ⚠️ → ✅.

## 6. Invariants added this session (don't break)

11. **legacy YAML path is dead** — no re-introduction of
    `loadPipelineConfig` / `PipelineConfig` / `FragmentRegistry`.
    New features must not grow YAML-engine-shaped helpers in
    `lib/config/`. CLAUDE.md §Retired areas lists the deletions.

12. **Safe-range constraint on AI-driven patches** —
    `proposePipelineFixWithAi` only attaches patches whose ops are
    all `update_stage_config`. Relaxing this requires a matching
    expansion in `isSafeRangePatch` + the synthesiser's allowedKeys
    filter + probably a new round of B4 review.

13. **compact_events are advisory** — writer.appendCompactEvent is
    called from the stream-pump; errors are swallowed inside the
    writer's NoopWriter branch. No caller should assume compact
    events are always present, especially on pre-T1 records.

14. **store_schema is strictly 1:1 with output ports** on the 4
    builtins. Editing an output port's type without updating the
    corresponding store_schema entry will now trigger
    STORE_SCHEMA_TYPE_MISMATCH at submit time — this is the
    intended behaviour. Keep the two in sync.

15. **fanout_element success attempts survive supersede** —
    migration-orchestrator's UPDATE explicitly carves them out. A
    future caller that writes its own supersede SQL must either
    replicate the exclusion or document why it shouldn't.
