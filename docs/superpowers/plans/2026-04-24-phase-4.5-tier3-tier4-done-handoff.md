| SHA | Subject |

| `9dc92a1` | docs(roadmap): Task 34 (script_execution_details) done — v1.8 history |

**Effect**: Script stage attempts now emit per-attempt sidecar rows
parallel to `agent_execution_details`. Columns captured:
`module_id`, `inputs_json`, `outputs_json`, `stdout`/`stderr`/`exit_code`
(pre-provisioned for future child_process executor modes — current
TS-function ScriptModule leaves them NULL), `error_message`/`error_stack`,
`duration_ms`, `started_at`/`ended_at`, `termination_reason` ∈
`{natural_completion, error, module_not_found, superseded}`.
Termination columns are NULLable (parallels agent sidecar) so
attempts that crash mid-flight are queryable via `ended_at IS NULL`.

The `prune-kernel-records` CLI was extended to delete the new sidecar
+ `migration_hints` (previously unhandled) scoped to pruned task_ids.

### Task 31 — A3 Store Schema (validator)

| SHA | Subject |
|---|---|
| `397ff4c` | feat(A3): pipeline-level store_schema + build-time drift validator |
| `5f55678` | docs(roadmap): A3 Store Schema validator landed — §6.3 status + v1.9 |

**Effect**: `PipelineIR.store_schema` optional top-level field. Each
entry has `type: string`, optional `description`, and
`produced_by: { stage, port }` pointing at a concrete output port.
New validator `validateStoreSchema` emits three diagnostic codes:
- `STORE_SCHEMA_STAGE_MISSING` — referenced stage not in ir.stages
- `STORE_SCHEMA_PORT_MISSING` — referenced port not an output of the stage
- `STORE_SCHEMA_TYPE_MISMATCH` — entry.type ≠ port.type (trimmed)

Wired into `KernelService.validate` (submit_pipeline path) and
`dryRunProposal` (propose_pipeline_change path). Type checking is
deferred to tsc (same convention as PortIR).

**Not shipped (user-intervention)**: migration of the 4 builtin IRs
(`smoke-test`, `tech-research-collector`, `tech-research-writer`,
`pipeline-generator`) to populate store_schema. That requires real
API calls through pipeline-generator.

### Task 33 — A4 dry_run_stage

| SHA | Subject |
|---|---|

**Effect**: New `debug/dry-run-stage.ts` module + MCP external tool.

caller supplies `pipelineVersion` + `stageName` + `inputs: Record<portName,

`STAGE_NOT_FOUND`, `STAGE_NOT_DRY_RUNNABLE` (gates), `MISSING_INPUT`,
`EXECUTOR_THREW`.

Schema: `stage_attempts.kind` CHECK extended with `'dry_run'`;
`PortRuntime.AttemptKind` extended likewise.

### Task 32 — A4 propose_pipeline_fix

| `b31deb1` | docs(roadmap): A4 propose_pipeline_fix landed — §6.4 status + v1.11 |

**Effect**: New `debug/propose-pipeline-fix.ts` module + MCP external
tool. Maps each hint in `TaskFailureReport` to a
`FixSuggestion { kind, targetStage, severity, description, rationale,
proposedPatch? }`. Pure-core `buildFixSuggestions(report, ir)` +
integration wrapper `proposePipelineFix({db, taskId, report})`.

stages that no longer exist in the current IR are filtered.

**Scope boundary (important)**: this is a RULE-BASED analyser. The
`proposedPatch` field is reserved for a future AI-driven layer that
synthesises a concrete IRPatch — that layer needs a real API call and
is NOT part of this autonomous session.

## 6. Important invariants (don't violate next session)

All invariants from the previous handoff still apply. New ones added
this session:

7. **script_execution_details** uses `ON DELETE RESTRICT` (not CASCADE)
   matching `agent_execution_details`. The prune CLI explicitly
   deletes sidecar rows before `stage_attempts`. Do not change this
   FK behaviour without also updating the CLI.

8. **dry_run_stage / replay_stage isolation**: both use an inert
   dispatcher. No future refactor of PortRuntime should allow port
   writes from a `kind='dry_run'` or `kind='replay'` attempt to
   reach an active XState machine. The `kind` column is the safety
   signal; any cleanup/prune/debug UI should filter appropriately.

9. **store_schema is OPTIONAL**: `PipelineIR.store_schema?: StoreSchema`.
   Validator runs only when the field is present. Existing pipelines
   (4 builtins) keep working without populating it. Do NOT make it
   required without first migrating the builtins.

10. **propose_pipeline_fix is rule-based**: its current output does
    not include `proposedPatch` values for any suggestion kind. A
    future AI-driven layer may fill that field. Existing suggestion
    consumers should treat `proposedPatch` as optional.
